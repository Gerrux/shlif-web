# CSV-экспорт и пакетная обработка — design

*2026-07-05. Триггер: пользователь сверился с полным списком требований к сервису и указал два
непокрытых пункта — экспорт метрик в CSV и пакетная (batch) обработка серии снимков. Остальные
пункты того же списка (предобработка, посегментная сегментация, классификация срастаний, детекция
талька, площади/доли, оверлей маски, таблица метрик в UI, PDF-протокол) уже реализованы текущим
пайплайном и вне скоупа этой спеки.*

## Согласованный скоуп

1. **Пакетная обработка** — только через веб-UI (не CLI/headless): drag-and-drop нескольких файлов
   разом, каждый обрабатывается независимо и **асинхронно в фоне**, не блокируя пользователя —
   он может в это время открыть и разобрать уже готовые снимки партии.
2. **CSV-экспорт** — как для одиночного снимка (аналог существующей кнопки PDF), так и сводный по
   всей партии (по одной строке на снимок).
3. Список job'ов партии переживает перезагрузку страницы/смену вкладки — персистентность на
   бэкенде (SQLite job store), не только в React state.

Вне скоупа: headless/CLI-батчинг, экспорт CSV из PDF-раза (отдельная кнопка вместо шаринга формата
с PDF), per-file выбор режима (крупный план/панорама) внутри одной партии, авторизация/раздельные
пользовательские истории (сервис и так без аутентификации, см. README).

## Бэкенд

### Схема `JobStore` (`backend/app/jobs/store.py`)

Таблица `jobs` получает три новые колонки поверх существующих (`id, mode, status, progress,
message, result`):

- `batch_id TEXT NULL` — генерируется на фронте (`crypto.randomUUID()`) один раз на партию
  загрузки и передаётся с каждым файлом; `NULL` для одиночной загрузки вне партии (обратная
  совместимость с текущим единичным flow — она не меняется).
- `filename TEXT NULL` — оригинальное имя файла (сейчас используется только для построения пути
  в `uploads_dir()`, в БД не попадает и клиенту не возвращается — нужно для отображения в
  галерее партии и для колонки в CSV).
- `created_at TEXT` — `strftime('%Y-%m-%dT%H:%M:%S')` в момент `create()`, для сортировки списка
  и для колонки в CSV.

Добавление колонок — аддитивный `ALTER TABLE jobs ADD COLUMN ...`, обёрнутый в try/except
`sqlite3.OperationalError` (колонка уже существует) в `__init__`, без миграции данных:
`backend/data/` в `.gitignore`, существующие локальные БД разработчиков не являются продакшн-данными.

### `POST /api/analyze` (`backend/app/api/analyze.py`)

Принимает необязательное поле формы `batch_id: str | None = Form(None)`. `store.create(mode,
batch_id=..., filename=...)` пишет все три новых поля. Поведение самого анализа (closeup/panorama
пайплайн) не меняется.

### `GET /api/jobs` (новый роут в `backend/app/api/jobs.py`, рядом с существующим `GET
/api/jobs/{jid}`)

Query-параметр `batch_id` (обязателен для этого маршрута в первой итерации — общий список всех
job'ов без фильтра не нужен, сервис однопользовательский и не должен превращаться в общий журнал).
Возвращает `list[JobRecord]` (те же поля, что и одиночный `GET /api/jobs/{jid}`, плюс `filename`,
`created_at`), отсортированный по `created_at`. Это то, что позволяет фронту восстановить галерею
партии после перезагрузки страницы, зная только сохранённый в `localStorage` `batch_id`.

`JobRecord` (`backend/app/schemas/jobs.py`) расширяется полями `filename: Optional[str]` и
`created_at: Optional[str]`.

### Конкурентность (`backend/app/jobs/runner.py`)

`JobRunner(max_workers=1)` → `max_workers` берётся из `SHLIF_JOB_WORKERS` (env, дефолт `4`). Один
воркер означает, что второй снимок партии физически не начинает обрабатываться, пока не
закончился первый — при партии из десятков снимков это и есть «тормозит пользователя». Пул из
нескольких воркеров даёт реальную фоновую параллельность для CPU-части пайплайна (GIL отпускается
внутри numpy/opencv/torch вызовов); GPU-инференс при этом просто сериализуется сам по себе на
уровне CUDA — безопасно, без падений, просто без ускорения этой части.

### Логирование параметров анализа (последний пункт исходного требования)

В `result`, уже сохраняемый в БД на каждый job, добавляется маленький под-объект `params`:
режим (`closeup`/`panorama`), какие чекпоинты моделей реально были в деле (переиспользуется
существующий `model_status()`/`gpu_available()` из `app/pipeline/loader.py`), ключевые пороги из
`cfg` (что уже вычисляется, просто сейчас никуда не пишется). Ничего нового по инфраструктуре —
просто фиксация уже существующих вычисляемых значений ради воспроизводимости; наружу в CSV не
выводится (не в списке колонок ниже), доступно через `GET /api/jobs/{jid}` для отладки/аудита.

## Фронтенд

### Загрузка партии

`DropZone`/`Welcome` (`frontend/components/upload/DropZone.tsx`, `Welcome.tsx`) получают атрибут
`multiple` и обработку `FileList` вместо одного `File`. Один файл → текущее поведение не меняется
ни на бит (сразу открывается одиночный workspace, `batch_id` не участвует). 2+ файла → фронт
генерирует один `batch_id`, отправляет `POST /api/analyze` на каждый файл **не дожидаясь ответа
предыдущего** (все уходят параллельно, каждый сразу возвращает свой `job_id`), и переключается на
новый экран — галерею партии.

### Галерея партии (новый компонент, например `frontend/components/batch/BatchGallery.tsx`)

Сетка карточек: превью — сразу `URL.createObjectURL` из выбранного `File` (доступно до всякого
ответа сервера), бейдж статуса (`queued/running/done/error`, тот же словарь, что уже есть в
`page.tsx` STATUS), обновляемый поллингом. Поллинг всех job'ов партии живёт на уровне галереи
(не внутри одиночного workspace), поэтому при переходе в конкретный снимок остальные продолжают
считаться и их статус не «замирает».

Клик по карточке (готовой или ещё считающейся) открывает существующий одиночный workspace —
`Corrector`/`PanoramaWorkspace`/`VerdictPanel` переиспользуются без изменений, только `jobId`
теперь приходит из выбранной карточки, а не из единственного `page.tsx`-стейта. Кнопка «← к
партии» возвращает в галерею.

### Персистентность на фронте

`batch_id` (и выбранный на партию режим) кладётся в `localStorage` при создании партии. При
монтировании приложения, если в `localStorage` есть незавершённый/недавний `batch_id`, фронт
дёргает `GET /api/jobs?batch_id=...` и восстанавливает галерею с актуальными статусами — часть
job'ов могла досчитаться, пока вкладка была закрыта.

### CSV-экспорт (без нового backend-эндпоинта)

Вся нужная информация уже приходит в JSON (`job.result.verdict.metrics`, `sort`, `ore_class`) —
CSV собирается чистой функцией на клиенте, без похода в сеть, что исключает рассинхрон между
PDF/JSON/CSV представлениями одних и тех же данных.

- Одиночный workspace: кнопка «Скачать CSV» рядом с существующей «Скачать протокол (PDF)»
  (`frontend/app/page.tsx`, там же где сейчас `reportUrl`) — одна строка с текущим job'ом.
- Галерея партии: кнопка «Скачать CSV (партия)», активна как только есть хотя бы один job со
  статусом `done`; строка на каждый job партии, включая `error`/ещё считающиеся — с пустыми
  метриками, но видимым статусом (чтобы в выгрузке было видно, что именно не досчиталось).

Колонки: `job_id, filename, mode, status, ore_class, sulfide_frac, magnetite_frac, matrix_frac,
talc_frac, talc_share_est, fine_share, confidence, undetermined_fraction, created_at`. Метрики —
сырые доли (0..1, без «%»), чтобы CSV сразу годился для расчётов в Excel/pandas. Метрика,
которой нет у конкретного job'а (набор отличается между closeup и panorama), — пустая ячейка.
Экранирование по RFC 4180 (запятые/кавычки/переводы строк в именах файлов и `text`).

## Тестирование

- Backend (`backend/tests/`): новый тест на `GET /api/jobs?batch_id=...` (фильтрация, сортировка
  по `created_at`, наличие `filename`/`created_at` в ответе); тест на увеличенный `max_workers`
  `JobRunner` — несколько job'ов реально выполняются конкурентно (не сериализуются строго один за
  другим); тест на аддитивную миграцию колонок (открыть существующую БД со старой схемой, убедиться
  что `JobStore.__init__` не падает и колонки появляются).
- Frontend (`frontend/tests/`): юнит-тест на чистую функцию CSV-сериализации по аналогии с
  существующими `encode.test.mjs`/`reducer.test.mjs` — форматирование чисел, пустые метрики,
  экранирование спецсимволов в именах файлов.
