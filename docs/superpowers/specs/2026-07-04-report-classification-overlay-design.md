# Отчёт: попиксельная маска обычные/тонкие срастания + тальк вместо тайловой закраски

- **Дата:** 2026-07-04
- **Статус:** согласовано в диалоге, готово к плану
- **Репозиторий:** `shlif-web`
- **Связано с:** `docs/superpowers/specs/2026-07-04-mask-editor-sidebar-design.md` (вкладки
  «Редактирование»/«Отчёт»), `docs/superpowers/specs/2026-07-04-panorama-closeup-unification-design.md`
  (единый `Corrector` и `verdict_from_masks_dict` для обоих режимов)

## 1. Проблема

ТЗ требует интерпретируемого результата: «Цветовая маска поверх исходного изображения (зелёный =
обычные срастания, красный = тонкие срастания, синий = тальк)». Сегодня этого нет нигде:

- Холст `Corrector.tsx` в любой вкладке рисует только «редакторские» цвета фаз (сульфид — латунь,
  магнетит — сталь, тальк — синий), никогда — классификацию обычные/тонкие.
- Легенда в `VerdictPanel.tsx` (три `.ptag`-чипа зелёный/красный/синий, добавлена в предыдущей
  итерации) — статичные подписи, не связанные ни с одним пикселем.
- Классификация обычные/тонкие (`_liberation_split` в `backend/app/shlif/analyze.py:33-52`, раскрытие
  зерна морфологическим открытием) **уже попиксельная и уже глобальная** (не по тайлам — см. §3), но
  её результат (`normal`, `fine`, булевы HxW-массивы) выбрасывается сразу после сведения к скалярным
  `normal_share`/`fine_share` — ни разу не сохраняется и не отдаётся по API ни для крупного плана, ни
  для панорамы (`backend/app/pipeline/masks.py:19-21`, `verdict_from_masks_dict`, обрезает до
  `{ore_class, text, metrics}`).
- Для панорамы холст сегодня строится поверх **уже тонированного** JPEG: `backend/app/pipeline/
  panorama.py:253` сохраняет `run["overlay"]` — картинку с полупрозрачной заливкой **каждого тайла
  целиком** одним цветом по решению RF-классификатора текстуры (`SORT_RGB`, тайловая, не попиксельная
  система) — в тот же файл (`images/{jid}.jpg`), который грузит холст как базовое фото
  (`Corrector.tsx:82`, `imageUrl(jobId)`). Крупный план сохраняет туда чистый снимок
  (`api/analyze.py:30-31`). Это и есть источник «тайловости», а не сама классификация обычные/тонкие.

## 2. Решения, принятые в диалоге

- Задача **только на холсте** (интерактивный `Corrector`) — панораму отдельно решать не нужно: с
  момента объединения режимов холст один на оба (`report-mode` дизайн ниже одинаково работает для
  panorama/closeup).
- В режиме «Отчёт» показываются **всегда все три цвета** (зелёный/красный/синий), независимо от
  глазков видимости сульфид/магнетит/тальк во вкладке «Редактирование» — те переключатели остаются
  чисто редакторским удобством.
- PDF-отчёт (`/api/report/{jid}.pdf`) — **вне рамок этой задачи**, отдельным заходом позже.
- Старая тайловая система закраски (`SORT_RGB`, блендинг `color_num`/`weight_den`/`talc_disp` в
  `overlay`) — **удаляется полностью**. RF-классификатор текстуры (`extract_features` +
  `clf.predict_proba` + `aggregate_section`), который питает карточку `sort` (не влияет на
  `ore_class`), **остаётся без изменений** — это другой, самостоятельный механизм (см. §4.2 дизайна
  объединения режимов), его нельзя путать с системой закраски тайлов.

## 3. Почему это не тайловая проблема (для панорамы)

Проверено по коду: `_liberation_split` вызывается **один раз глобально**, на уже собранном (core-crop,
без нахлёста, без фезеринга — see `_assemble_masks`, `panorama.py:71-110`) полноканвасном
`assembled["sulfide"]` (`panorama.py:246-248` → `verdict_from_masks_dict` → `verdict_from_masks` →
`_liberation_split`, `analyze.py:55-90`), с радиусом раскрытия, посчитанным от `long_side` всего
холста (`analyze.py:60-61`), а не от размера тайла. Швов/блочности в самой классификации нет и не
может быть — это отдельно от системы закраски тайлов, которую удаляем.

Более того: «тайлы» в `_run_panorama` (место, где живёт закраска) — это не отдельно снятые и сшитые
фото, а окна анализа поверх **одного уже целиком декодированного массива** (`arr`, `load_working_array`,
`tiling.py:59-64`). Базовое фото (`base`/`edit`, `panorama.py:128,134`) собирается один раз целиком,
до применения тайловой тонировки — швов на самом фото нет и не будет, убирать/чинить сшивку не нужно
(см. §5).

## 4. Дизайн

### 4.1 Бэкенд — новый персистентный слой `intergrowth`

Новая функция-конвенция (по образцу уже существующей `phase_label_map`), `backend/app/pipeline/
masks.py`:

```python
def intergrowth_label_map(normal: np.ndarray, fine: np.ndarray) -> np.ndarray:
    im = np.zeros(normal.shape, np.uint8)   # 0 = не сульфид (магнетит/матрица/тальк)
    im[fine.astype(bool)] = 2               # тонкие срастания
    im[normal.astype(bool)] = 1             # обычные срастания
    return im
```

`verdict_from_masks_dict` (`masks.py:19-21`) начинает прокидывать готовую попиксельную карту дальше,
вместо того чтобы её выбрасывать:

```python
def verdict_from_masks_dict(sulfide, magnetite, matrix, talc, cfg) -> dict:
    v = verdict_from_masks(sulfide, magnetite, matrix, talc, cfg)
    return {"ore_class": v["ore_class"], "text": v["text"], "metrics": v["metrics"],
            "intergrowth": intergrowth_label_map(v["normal"], v["fine"])}
```

Это единственная точка, которую сейчас используют **и** первичный анализ панорамы
(`panorama.py:247-248`), **и** эндпоинт сохранения/пересчёта масок (`api/masks.py:48`, общий для обоих
режимов) — правка одного места чинит оба сценария правки/пересчёта.

**Важно:** `"intergrowth"` — сырой `np.ndarray`, не JSON-сериализуемый. Оба вызывающих места обязаны
`pop()` его из словаря **до** того, как остаток словаря уйдёт в JSON-ответ (иначе — либо падение
сериализации, либо утечка постороннего поля в `Verdict`). Класть в `Verdict` наружу не нужно —
попиксельные данные едут отдельным PNG-слоем, как `phases.png`/`talc.png`, а не полем JSON.

**Точки вызова:**

- `backend/app/pipeline/closeup.py`, `analyze_closeup` — `res.masks["normal"]`/`["fine"]` уже
  посчитаны (`analyze.py:158-159`), просто не форвардятся в возвращаемый словарь. Добавить
  `"intergrowth": masks.intergrowth_label_map(m["normal"], m["fine"])` в dict, который уходит в
  `persist_editor_artifacts` (нулевая доп. вычислительная стоимость — данные уже есть).
- `backend/app/pipeline/panorama.py`, `analyze_panorama` — `verdict.pop("intergrowth")` сразу после
  вызова `verdict_from_masks_dict` (до попадания `verdict` в возвращаемый JSON-словарь), затем —
  **тот же ресайз до разрешения `edit`** (`cv2.resize(..., interpolation=cv2.INTER_NEAREST)`), что уже
  применяется к `sulfide_small`/`magnetite_small`/`talc_small` (`panorama.py:257-262`), и добавить
  результат в словарь для `persist_editor_artifacts` (строки 269-274).
- `backend/app/api/masks.py`, `save_masks` — **важное уточнение, не очевидное с первого взгляда**:
  `phases.png`/`talc.png` тут пишутся **до** апскейла к native (строки 36-37, разрешение редактора —
  то, что реально загрузил браузер), а `verdict_from_masks_dict` на строке 48 вызывается **после**
  апскейла (строки 40-44, если панорама — native-разрешение, для панорамы это другой, больший размер;
  для крупного плана `native` отсутствует, апскейл — no-op, тут проблемы нет вообще). Если писать
  `intergrowth.png` из результата строки 48 как есть, для панорамы он получится **в native-разрешении,
  а `phases.png`/`talc.png` — в разрешении редактора** — разные размеры одного и того же job. Фронтенд
  тянет все три PNG через `pngToArray(url, w, h)` с одним и тем же `w,h` (`ctx.drawImage(img,0,0,w,h)`
  растянет несовпадающий по размеру PNG без ошибки, но билинейно — а метка класса 0/1/2 при
  интерполяции испортится на границах). Значит: сохранить `orig_h, orig_w = pm.shape` **до** блока
  апскейла; после `v.pop("intergrowth")` — если его форма не совпадает с `(orig_h, orig_w)`, досжать
  `cv2.resize(intergrowth, (orig_w, orig_h), interpolation=cv2.INTER_NEAREST)` — и только потом
  писать `intergrowth.png`. Это даёт вычисление на точном (native) разрешении для метрик (как и
  сегодня для `normal_share`/`fine_share`) и хранение в разрешении редактора (как `phases.png`/
  `talc.png`) — те же данные, два представления, ровно как уже сделано для панорамы в §4.1 выше
  (`sulfide_small`/`magnetite_small`/`talc_small`).

`persist_editor_artifacts` (`masks.py:66-78`) — добавить одну строку:
`(md / "intergrowth.png").write_bytes(encode_png_gray(r["intergrowth"]))`, та же кодировка, что у
`phases.png` (значения 0/1/2 как есть в пиксель, не 0/255 — как `phase_map`, не как булева
`talc`-маска).

### 4.2 Бэкенд — новый эндпоинт слоя

`backend/app/api/masks.py`, `get_mask` (строка 14): добавить `"intergrowth"` в множество разрешённых
слоёв — `{"phases", "talc", "intergrowth"}`. Отдаётся тем же путём: `GET /api/masks/{jid}/
intergrowth.png`.

### 4.3 Бэкенд — убрать тайловую закраску панорамы

В `backend/app/pipeline/panorama.py`, `_run_panorama`:

- Убрать: `SORT_RGB`, `TALC_RGB` (константы, строки 44-45); `sort_alpha` (148); аккумуляторы
  `color_num`/`weight_den`/`talc_disp` (139-141) и их заполнение (196-199, 202-205); финальный блендинг
  `overlay = base.copy(); ... out[talc_disp] = ...` (211-218); переменные `dx1,dy1` и связанный с ними
  guard (170-172) — они нужны были только для геометрии закраски, `low_conf_zones` использует только
  `dx0,dy0` + `bx,by` (184-186), не трогать.
- Оставить без изменений: RF-классификатор (`extract_features`, `clf.predict_proba`, `records`,
  `aggregate_section`, `sort_proba`/`sort_top`) — питает карточку `sort`, к закраске не относится;
  `ore_density`/`bright_thr`/`n_ore`/`n_matrix`/`n_tiles`; неопределённость/`low_conf_zones`; `dg`/talc
  где он относится к метрикам, а не к рисованию.
- `base = edit.astype(np.float32)` — убрать (был нужен только для блендинга); использовать `edit`
  (уже `uint8`) напрямую.
- Возвращаемый словарь `_run_panorama` (сейчас строки 220-227) — убрать ключ `"overlay"`, оставить
  `"edit_rgb": edit` как есть (уже ровно то самое чистое изображение).
- `analyze_panorama` (строка 253): `Image.fromarray(run["overlay"]).save(...)` →
  `Image.fromarray(run["edit_rgb"]).save(...)`.
- `analyze_panorama`, финальный возвращаемый словарь (строка 284): убрать `"overlay_url"` — поле
  больше не нужно (фронтенд его уже не читает, `imageUrl(jobId)` указывает на тот же путь и теперь
  всегда отдаёт чистое фото). `frontend/lib/api/types.ts`: убрать `overlay_url?` из `AnalyzeResult` —
  не используется нигде ни в одном компоненте (проверено).

### 4.4 Фронтенд — рендер на холсте (`Corrector.tsx`)

- `frontend/lib/api/client.ts`: `maskUrl(id, layer: "phases" | "talc" | "intergrowth")`.
- Новые константы (значения — те же, что уже определены в бэкенде,
  `backend/app/shlif/phases.py:45-46`, для консистентности «маска в редакторе» ↔ «маска в отчёте»):
  `const NORMAL_RGB: [number, number, number] = [63, 174, 107];` (зелёный),
  `const FINE_RGB: [number, number, number] = [224, 85, 78];` (красный). `TALC_RGB` уже существует
  (`[79, 143, 240]`) и переиспользуется как есть.
- Новый `useRef<Uint8Array | null>(null)` — `intergrowthRef`; загружается в том же месте, что
  `phasesGray`/`talc` (в `useEffect` по `[jobId, w, h]`):
  `intergrowthRef.current = await pngToArray(maskUrl(jobId, "intergrowth"), w, h);`
- Новый `useRef` — `sideTabRef` (тот же паттерн, что `visRef`/`maskAlphaRef`: `useState` + синхронный
  `.current =` на каждый рендер), чтобы `composePixel` (обычная функция вне реактивности) мог читать
  активную вкладку.
- `composePixel` — ветвление по `sideTabRef.current`:
  - `"report"`: **всегда** (не читая `visRef`) — если `intergrowth[i] === 1` → блендить `NORMAL_RGB`
    при `a = maskAlphaRef.current`; если `intergrowth[i] === 2` → блендить `FINE_RGB`; если `tc[i]` →
    блендить `TALC_RGB`. Магнетит/матрица в отчёте отдельным цветом не красятся (их «не сульфид,
    не тальк» — 0 в `intergrowth`) — ровно 3 цвета, как в ТЗ.
  - `"edit"`: сегодняшнее поведение без изменений (сырые цвета фаз, с учётом `visRef`).
- Добавить `sideTab` в зависимости уже существующего эффекта полной перерисовки
  (`Corrector.tsx:101-107`) — переключение вкладки должно сразу перерисовать холст новой цветовой
  схемой.
- После успешного `save()` (`Corrector.tsx:248-257`) — перезапросить `intergrowth`
  (`intergrowthRef.current = await pngToArray(maskUrl(jobId, "intergrowth"), w, h);`) и
  `requestDraw()`, аналогично тому, как `onVerdict(v)` уже обновляет вердикт — правки сульфида/
  магнетита меняют классификацию, отчёт должен сразу её отражать.
- Тумблеры «Слои» (сульфид/магнетит/тальк, вкладка «Редактирование») **не меняются** — они управляют
  только редакторским рендером, к отчётному не имеют отношения (см. §2).

## 5. Риски / известные ограничения

- **Несовпадение разрешений в `save_masks`** (см. §4.1) — самый реальный риск во всей задаче: если
  забыть досжать `intergrowth` до разрешения редактора для панорамы после пересчёта, `intergrowth.png`
  окажется другого размера, чем `phases.png`/`talc.png`, и холст растянет его с билинейной
  интерполяцией — метка класса 0/1/2 испортится на границах обычные/тонкие. Явно проверить в плане
  реализации отдельным тестом (см. §7).
- Ресайз `intergrowth` для панорамы (nearest-neighbor, значения 0/1/2) — тот же паттерн, что уже
  используется и проверен для `phase_map`/`talc` панорамы; отдельного риска не добавляет.
- Удаление `overlay_url`/`"overlay"` — чисто уборка мёртвого поля; фронтенд его не читает (проверено
  прямым поиском по `frontend/`), риск регрессии отсутствует.
- Точные RGB-цвета зелёного/красного в легенде (`VerdictPanel.tsx`, CSS-токены `--phase-normal`/
  `--phase-fine`) и в новых канвас-константах (`[63,174,107]`/`[224,85,78]`, взятые из бэкендового
  `phases.py`) не обязаны быть побитово идентичны (один — через oklch/CSS, другой — сырой RGB для
  канваса) — это тот же класс несоответствия, что уже принят для `maskAlpha`/`TALC_RGB` в предыдущей
  фиче; важно совпадение по смыслу (зелёный/красный/синий), не по хешу цвета.

## 6. Вне рамок (YAGNI)

- PDF-отчёт (`backend/app/pipeline/report.py`) — не трогаем.
- Изменение самого алгоритма `_liberation_split`/порогов классификации — не трогаем, только
  прокидываем уже готовый результат.
- Настраиваемость (тумблер видимости) отчётной маски — всегда все 3 цвета, без настроек (§2).

## 7. Проверка

- Бэкенд: юнит-тест на `intergrowth_label_map` (простая функция — вход/выход на синтетических
  массивах); юнит/интеграционный тест, что `GET /api/masks/{jid}/intergrowth.png` отдаёт корректный
  слой после `/analyze` для обоих режимов и после `POST /masks/{jid}` (пересчёт) — значения меняются
  вместе с правками сульфида; **отдельный тест именно на совпадение разрешений** — после
  `POST /masks/{jid}` на панорамном job'е декодировать `intergrowth.png` и сравнить его `shape` с
  `phases.png`/`talc.png` того же job'а (см. риск в §5); тест, что `verdict_from_masks_dict`'s
  `"intergrowth"` не протекает в JSON-ответ ни одного из двух эндпоинтов, которые его используют.
- Фронтенд: юнит-тест не нужен (то же ограничение, что и раньше — `Corrector.tsx` не покрыт
  компонентными тестами, DOM/canvas-логика проверяется сборкой + вручную).
- Ручная проверка: загрузить и крупный план, и панораму, открыть «Отчёт» — увидеть зелёный/красный
  (+ синий тальк, если есть) вместо латуни/стали; переключиться на «Редактирование» — вернулись сырые
  цвета фаз; поправить сульфид, сохранить, вернуться в «Отчёт» — маска обновилась; убедиться, что
  панорамное фото под маской больше не в тайловых пятнах.
