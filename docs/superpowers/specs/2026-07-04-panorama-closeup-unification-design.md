# Убрать режимы «крупный план / панорама»: автоопределение, единый вердикт, редактируемые маски для обоих

- **Date:** 2026-07-04
- **Status:** draft — pending review
- **Repo:** `shlif-web`
- **Relates to:** `docs/superpowers/specs/2026-07-04-shlif-web-unified-service-design.md` (§11.5 explicitly
  deferred full panorama correction to a later iteration — this spec is that iteration)

---

## 1. Context & goal

Сегодня режим анализа (`крупный план` / `панорама`) — ручной тумблер на фронтенде, отправляемый как
`mode` в `POST /analyze` (`frontend/app/page.tsx:13,77-82`, `backend/app/api/analyze.py:23`). Пайплайны
за этим тумблером устроены по-разному:

- **Крупный план** (`pipeline/closeup.py`) — один проход `analyze_image` по всему кадру →
  `verdict_from_masks` даёт `{sulfide_frac, magnetite_frac, matrix_frac, talc_frac, normal_share,
  fine_share, confidence, talc_share_est}`, плюс `sort` (RF-классификатор по текстуре всего кадра),
  плюс редактируемые маски (`Corrector.tsx`) — площадь считается как `mask.sum()/total_pixels`, то
  есть честно относительно всего кадра.
- **Панорама** (`pipeline/panorama.py`) — тайлит скан (нативные тайлы 2048px, оверлап из конфига),
  классифицирует КАЖДЫЙ рудный тайл RF-классификатором, `ore_class` вердикта — argmax
  ore-density-взвешенного среднего вероятностей по тайлам (`aggregate_section`). Маски **не
  редактируются** (`PanoramaWorkspace.tsx` — только зум/пан, коммент в коде: «правка масок панорам
  требует поддержки бэкенда»).

Пользовательский запрос: убрать ручной выбор режима вовсе — определять его автоматически по размеру
изображения (нужно ли тайлирование для скармливания в модели, или можно обработать целиком за один
проход), и сделать так, чтобы результат (площади минералов относительно ВСЕГО изображения, не по
тайлам) и возможность редактировать маски были одинаковы в обоих случаях.

Разведка реального датасета (`hakaton_nornikel/sumple_dataset`) подтверждает чистое разделение по
размеру и заодно вскрыла существующий баг:

| Папка | n файлов | Диапазон разрешений |
|---|---|---|
| `Фото руд по сортам. ч1` | 220 | ровно 3.9 МП (2272×1704) |
| `Фото руд по сортам. ч2` | 1002 | 0.2–26 МП, p95≈21.6 МП |
| `Панорамы` | 14 | 126.5–574 МП |

Между 26 МП и 126.5 МП в выборке нет ни одного файла — порог в этом диапазоне безопасен.

**Существующий баг, который эта работа заодно чинит:** `panorama.py:136` считает
`talc_frac = talc_px / (talc_px + matrix_px)` — рудные пиксели (сульфид/магнетит) вообще не входят в
знаменатель, а `talc_px`/`matrix_px` суммируются по тайлам, которые перекрываются на `overlap` px
(128 по конфигу), т.е. полосы нахлёста считаются дважды. Плюс пропущенные «пустые» тайлы
(`tiling.py: _is_empty`) вообще не добавляют свою площадь ни в `matrix_px`, ни куда-либо ещё —
недосчёт. Новый дизайн считает всё через ту же `mask.sum()/total_pixels`, что и крупный план,
поэтому оба бага пропадают как следствие, а не как отдельный патч.

## 2. Non-goals

- Настоящая склейка/регистрация панорамы (`cv2.Stitcher`, `findHomography`, ORB/SIFT) — вход
  по-прежнему один уже готовый растр, как сегодня.
- Полноразрешающий (нативный, гигапиксельный) интерактивный редактор масок — редактирование остаётся
  на уменьшенной копии (см. §5), апскейл при пересчёте. Панорамный просмотр «в максимальном разрешении
  с зумом» не блокируется, но это не про редактирование.
- Изменение алгоритма сегментации (`segment_phases`, `detect_talc`, `dark_gray_phase`) — переиспользуется
  как есть, просто вызывается по-другому (по тайлам + сборка вместо одного прохода).
- Изменение УЖЕ имеющегося порога декодирования в память `tiling.max_pixels` (150 МП, draft-decode
  выше него) — новый порог `direct_max_pixels` (50 МП) решает другую задачу (нужно ли тайлить вообще) и
  не заменяет его.

## 3. Автоопределение режима

Новое поле конфига:

```yaml
tiling:
  direct_max_pixels: 50000000   # ≤ это: один проход без тайлинга (сегодняшний «крупный план»)
```

На входе `POST /analyze` (`backend/app/api/analyze.py`): вместо `mode: str = Form("closeup")` —
сервер сам читает размеры изображения (дешёво, без полного декода) и сравнивает `w*h` с
`tiling.direct_max_pixels`:

- `≤ порога` → сегодняшний `analyze_closeup` путь, **без изменений** (включая
  `im.thumbnail((2400,2400))` на входе — эта строка и всё что после неё не трогается).
- `> порога` → новый тайловый путь (§4).

`mode` как поле в БД (`jobs.mode`, `jobs/store.py`) остаётся — просто это теперь производный факт
(«как определили», не то, что прислал клиент), пишется в job после детекта. API-ответ и PDF-протокол
(`report.py:84`, «Режим: …») продолжают показывать это же поле, просто как результат детекта, а не
пользовательский выбор.

## 4. Тайловый путь: сборка единой маски на весь снимок

Ключевая идея: не изобретать отдельный «вердикт панорамы» — **переиспользовать
`verdict_from_masks(sulfide, magnetite, matrix, talc, cfg)`** (`shlif/analyze.py:41`, уже общая точка
входа) для панорамы точно так же, как для крупного плана, подав ей на вход не один проход, а маски,
собранные из тайлов. Тогда идентичность структуры результата — следствие того, что это буквально одна
и та же функция, а не совпадение форматов, которое может снова разъехаться.

**Алгоритм сборки (per tile → core-crop → один канвас):**

1. `Wt, Ht, factor = tile_grid(path, cfg.tiling)` — как сегодня (рабочий холст после
   draft-decode, если снимок больше `max_pixels`).
2. Выделяются булевы канвасы на весь холст: `sulfide`, `magnetite`, `matrix`, `talc`, `dg`
   (дисперсная тёмно-серая фаза для `talc_share_est`, как `dark_gray_phase` в closeup).
3. Для каждого тайла из `iter_tiles`:
   - если `tile.empty` (пуст, только матрица — эвристика `_is_empty` уже есть) — пишем `True` в
     `matrix` для **ядра** тайла (см. ниже) и пропускаем тяжёлые вычисления, как сегодня. Отличие от
     сегодняшнего кода: сегодня пустые тайлы вообще не учитываются в площади — теперь учитываются, это
     часть починки бага.
   - иначе: `preprocess` → `segment_phases(...).labels` (уже даёт 0/1/2 = матрица/магнетит/сульфид,
     сегодня `panorama.py:91` использует только `==0`, остальное выбрасывается) → `phase_label_map`
     (та же функция `masks.py:8`, что и в closeup) → `detect_talc` → `dark_gray_phase` → пишем core
     каждой маски в общий канвас.
4. **Core-crop, не feather:** соседние тайлы перекрываются на `overlap` px для контекста сегментации
   у края (чтобы не резать минеральное зерно ровно по границе тайла), но при сборке маски **берём
   только неперекрывающуюся часть** каждого тайла — `[x, x+step)` по каждой оси, где
   `step = tile - overlap` (шаг, с которым `iter_tiles` и так сдвигает окно). Последний тайл в
   строке/столбце пишет до истинного края холста (шаг не обязан ровно делить размер холста — здесь
   нужно аккуратно свести до `Wt`/`Ht`, иначе останется полоса-пробел). Никакого фезеринга/бленда для
   масок — каждый пиксель приписан ровно одному тайлу, поэтому суммы `mask.sum()` по всему канвасу
   корректны без двойного счёта. (Фезеринг остаётся как есть, но только для косметического
   preview-оверлея, не для счёта.)
5. После сборки: `verdict = verdict_from_masks(sulfide, magnetite, matrix, talc, cfg)` — тот же
   вызов, что в closeup. `metrics["talc_share_est"] = dg.mean()` по тому же собранному канвасу.

Итог: `verdict.metrics` для панорамы содержит ровно те же ключи, что для крупного плана
(`sulfide_frac, magnetite_frac, matrix_frac, talc_frac, normal_share, fine_share, confidence,
talc_share_est`), посчитанные как доля от площади **всего** изображения.

### 4.1 Осознанное изменение семантики: `ore_class` становится правилом, а не голосованием классификатора

Сегодня `verdict.ore_class` панорамы — это `argmax` ore-density-взвешенного среднего вероятностей
RF-классификатора по рудным тайлам (`aggregate_section`, `panorama.py:41,120-121`). Для крупного плана
`ore_class` — результат ПРАВИЛА (`talc_frac`/`fine_share` против порогов, `verdict_from_masks:50-61`),
а RF-классификатор — отдельная, параллельно показываемая карточка `sort` (текстурная эвристика, не
влияет на `ore_class`).

Раз результаты должны совпадать по структуре и смыслу — панорама тоже переходит на **правило** как
источник `ore_class` (шаг 5 выше), а RF-классификатор remains ровно тем же вспомогательным `sort`
(§4.2), как в closeup. Это меняет сегодняшнее поведение панорамы (сейчас именно классификатор решает
`ore_class`) — явно выношу этот пункт на подтверждение при ревью спеки, а не прячу как деталь
реализации.

### 4.2 `sort`-карточка (RF-классификатор)

Сегодняшний механизм — классификация КАЖДОГО рудного тайла + ore-density-взвешенное усреднение
(`aggregate_section`) — вероятно намеренно тюнингован под то, что признаки классификатора (текстурные,
локальные) теряют смысл при наивном downscale огромного скана в один маленький кадр. Менять сам
механизм классификации — отдельный, более рискованный шаг, не запрошенный явно. Поэтому:

- **Механизм классификации не меняется** — тот же per-tile `extract_features` + `clf.predict_proba` +
  `aggregate_section`.
- **Меняется только форма ответа**: сегодня результат зарыт в `verdict.metrics.sort_proba`
  (`panorama.py:157`). Переносится в top-level `sort = {"classes": {...}, "top": "..."}` — тот же ключ и
  форма, что `_sort_card` в closeup (`closeup.py:11-19,45`), просто наполненный из
  `aggregate_section`-агрегата вместо одиночного вызова классификатора.

### 4.3 Неопределённость (`confidence`/`low_conf_zones`) и `superpixels`/`darkness` — на уменьшенной копии, не по тайлам

`ensemble_uncertainty` уже сегодня в closeup считается на уменьшенной копии (`_UNC_MAX_SIDE=1024`,
`closeup.py:9,21-30`) — комментарий в коде явно говорит, что доля (`undetermined_fraction`)
устойчива к масштабу. `build_superpixel_map`/`build_darkness_map` (`masks.py:21-26`) сегодня тоже
гоняются по всему (уже небольшому) кадру целиком.

Сшивать SLIC-суперпиксели или confidence-карту по тайлам (с оффсетом ID, склейкой швов) — не нужно:
эти артефакты нужны только редактору и визуализации, не участвуют в площадном счёте. Проще и надёжнее
посчитать их **один раз** на единой уменьшенной копии всего снимка (§5), той же функцией, что и в
closeup — не по тайлам вообще.

## 5. Разрешение для редактирования — отдельное от разрешения для анализа

**Уточнение к более раннему ответу в диалоге:** ранее было сказано «редактирование на копии ≤50 МП» —
это неверно, есть уже проверенная в бою константа: closeup сегодня анализирует и редактирует на одной и
той же копии ≤2400×2400 (~5.76 МП, `analyze.py:34`, `im.thumbnail((2400,2400))`). Эта константа и
используется как разрешение редактора для **обоих** путей — не 50 МП. `direct_max_pixels` (50 МП)
остаётся отдельным порогом только для решения «тайлить или нет» (§3).

Итого два независимых разрешения:

- **Разрешение анализа** — нативное (после существующего `max_pixels`-ограничения памяти), даёт
  точность подсчёта площади; для снимков ≤50 МП совпадает с разрешением редактора (thumbnail — уже
  no-op для большинства снимков в датасете, см. таблицу в §1).
- **Разрешение редактора** — единая уменьшенная (≤2400×2400) RGB+label-копия всего снимка (для
  панорамы — уменьшенная копия собранных из §4 масок), на ней же считаются superpixels/darkness/confidence
  (§4.3). Одна и та же UX-производительность редактирования в обоих случаях, как сегодня у closeup.

**Пересчёт после правки:** `POST /masks/{jid}` декодирует присланные PNG (в разрешении редактора),
апскейлит (nearest-neighbor — корректно для целочисленных меток класса) до разрешения анализа, и
вызывает `verdict_from_masks` на апскейленных масках. Для closeup разрешение анализа и редактора
совпадают → апскейл — no-op, поведение не меняется. Один код без ветвления по режиму.

## 6. API и фронтенд

### 6.1 Backend

- `POST /api/analyze`: убрать `mode: str = Form(...)`. Сервер сам решает режим (§3) и запускает
  соответствующую ветку. Ответ обеих веток — одна и та же форма:
  `{mode, verdict: {ore_class, text, metrics}, sort: {classes, top}, text, size: [w,h],
  low_conf_zones, overlay_url?}`. (`overlay_url` — опциональный превью-оверлей; сегодня есть только у
  панорамы, но раз редактор теперь общий, скорее всего не нужен отдельно от `images/{id}.jpg` — решить
  при реализации, не меняет структуру вердикта.)
- `POST /api/masks/{jid}`: без изменений в контракте, кроме апскейла перед `verdict_from_masks_dict`
  (§5) — панорама теперь тоже попадает в этот путь.
- `GET /api/masks/*`, `GET /api/maps/*`: без изменений — просто теперь пишутся и для панорамного job.

### 6.2 Frontend

- `frontend/app/page.tsx`: убрать `MODES`/`onMode`/тумблер «Крупный план · Панорама»
  (`page.tsx:13,77-82`) — загрузка без выбора режима. Информационную плашку «Образец: …» брать из
  `result.mode`, который теперь приходит с сервера, а не из локального `useState`.
- Убрать ветвление `mode === "closeup" ? <Corrector/> : <PanoramaWorkspace/>` (`page.tsx:89-92`) —
  всегда `<Corrector/>`, раз маски теперь редактируемы в обоих случаях и обе ветки отдают одинаковые
  layer/map артефакты. `PanoramaWorkspace.tsx` (34 строки, чистый zoom/pan-просмотрщик без правки,
  коммент в коде прямо говорит «правка требует поддержки бэкенда» — эта поддержка появляется) можно
  удалить; если нужен зум поверх исходного (не уменьшенного) оверлея для визуальной проверки перед
  редактированием — это отдельная, не блокирующая эту работу доработка `Corrector`.

## 7. Риски / известные ограничения

- **Память при сборке масок:** для 570 МП скана (после `max_pixels=150М` draft-decode — реальный
  рабочий холст меньше) 4–5 булевых канваса + рабочий RGB-массив — заметный, но разовый и ограниченный
  объём памяти на job; не блокер, но стоит проверить пиковое потребление на самом большом файле выборки
  (`Панорамы/16.jpg`, 574 МП) при реализации.
- **Граница последнего тайла в строке/столбце:** шаг `tile-overlap` не обязан ровно делить размер
  холста — код сборки должен явно дотягивать core последнего тайла до истинного края, иначе полоса
  пикселей у правого/нижнего края останется непокрытой ни одним тайлом.
- **Семантика `ore_class` меняется для панорамы** (§4.1) — было решение классификатора, станет решение
  правила. Явно вынесено на подтверждение при ревью спеки.

## 8. Тестирование

- Юнит: сборка core-crop канваса на синтетическом наборе тайлов (не должна давать двойной счёт/пробелы
  на границах) — сравнить `mask.sum()` собранного канваса с прямым подсчётом на эталонном полном
  массиве.
- Юнит: `verdict_from_masks` возвращает одинаковый набор ключей metrics для closeup- и
  panorama-входа.
- Юнит: авто-детект режима — таблица размеров вокруг порога (49.9 МП / 50.1 МП) даёт ожидаемую ветку.
- Интеграционный: реальный файл из `Панорамы/` целиком через `/analyze` → `/masks` (правка) →
  пересчитанный вердикт с ожидаемой формой; и файл из `Фото руд по сортам.` — что поведение closeup
  бит-в-бит не изменилось (регрессия на существующих golden-результатах, если такие есть, иначе —
  снимок метрик до/после рефактора panorama-обвязки, closeup код не трогается).
- Ручная проверка (`verify` skill): загрузить и панораму, и крупный план через фронтенд без тумблера,
  убедиться что кнопка «Доработать маски» доступна в обоих случаях и пересчёт после правки работает.
